# Décisions techniques

Décisions non couvertes (ou différentes) du PRD, avec justification courte. Une ligne par décision, les plus récentes en bas.

- **2026-07-04 — Template Vite officiel actuel (Vite 8, TypeScript 6, oxlint)** : le scaffold officiel `create-vite` livre aujourd'hui oxlint comme linter au lieu d'ESLint ; on garde ce standard plutôt qu'une config ESLint maison, Prettier ajouté pour le formatage.
- **2026-07-04 — Tailwind CSS v4 via plugin `@tailwindcss/vite`** : version actuelle de Tailwind ; la config se fait en CSS (`@custom-variant` pour le mode sombre par classe), il n'y a plus de `tailwind.config.js` comme décrit dans le PRD (écrit pour v3).
- **2026-07-04 — React Router v7 (package `react-router`)** : version actuelle, import direct depuis `react-router` (le package `react-router-dom` est déprécié en v7).
- **2026-07-04 — SRS : SM-2 à 3 boutons** : « À revoir » (qualité 2, réinitialise l'intervalle), « Difficile » (qualité 3), « Réussi » (qualité 5). Trois boutons max pour un ado — moins de friction que l'échelle 0-5 complète d'Anki.
- **2026-07-04 — Carte ratée re-mise en fin de file de la session** : feedback non punitif (PRD §17) — on retravaille le mot immédiatement au lieu d'attendre le lendemain.
- **2026-07-04 — Sens de révision M0 : NL→FR uniquement (reconnaissance)** : le sens production FR→NL (plus difficile, plus proche de l'examen) sera ajouté quand le contenu réel arrivera, pour ne pas doubler la charge dès le premier jour.
- **2026-07-04 — Plafonds de session : 20 révisions + 8 nouveaux mots** : valeurs classiques type Anki, à ajuster à l'usage ; sessions courtes = sentiment de complétion fréquent (PRD §17).
- **2026-07-04 — PWA (`vite-plugin-pwa`) reportée à M1** : M0 = squelette minimal utilisable ; le manifest + service worker arrivent avec la premiere mise en prod réelle.
- **2026-07-04 — XP : réussi = 10 × difficulté, difficile = moitié, raté = 0 (jamais négatif), bonus +20 si session terminée, plafond 300/session** : simple, non punitif (PRD §17), anti-farming (PRD §8) ; la pondération par modalité viendra quand d'autres modalités existeront.
- **2026-07-04 — Streak M1 : ≥ 1 session terminée/jour (seuil minutes paramétré à 0)** : le PRD (§8) fixe ≥ 1 h/jour, mais avec le seul module vocab une heure est irréaliste — exiger 1 h rendrait le streak inatteignable et démotiverait, l'inverse du but. Le seuil passera à 60 min quand grammaire + écoute existeront (M2/M3). La fonction accepte déjà `minMinutesPerDay`.
- **2026-07-04 — Schéma Supabase : `households` + `profiles` au lieu d'une table `users` unique** : Supabase Auth gère déjà `auth.users` ; `profiles` (1-1 avec auth.users, rôle parent/enfant) + `households` (lien foyer) permettent les policies « le parent lit le foyer » du PRD Annexe A. `content_items.household_id` null = contenu global de départ.
- **2026-07-04 — Grants explicites dans la migration** : les défauts Supabase actuels ne donnent plus aucun privilège DML aux rôles API (`authenticated` n'avait que TRUNCATE/REFERENCES/TRIGGER) ; on accorde le strict nécessaire par table, pas d'update/delete sur les tables append-only (sessions, xp_ledger), rien à `anon`.
- **2026-07-04 — RPC `create_household_with_profile` (security definer) pour l'onboarding** : impossible de récupérer l'id du foyer via `INSERT ... RETURNING` avant que le profil existe (la policy SELECT ne montre le foyer qu'à ses membres) ; la RPC crée foyer + profil atomiquement. Rejoindre un foyer = connaître son uuid (fait office d'invitation, non devinable).
- **2026-07-04 — Tests RLS dans `supabase/tests/rls-check.mjs` (node, stack local Docker)** : 18 vérifications d'isolation (enfant/parent/autre foyer) exécutables à chaque évolution du schéma ; hors Vitest car dépendant de Docker.
- **2026-07-04 — Comptes du foyer : parent = email + mot de passe ; élève = vrai compte Supabase avec email en plus-addressing suggéré (`x+klaar@y`) et code simple (6 caractères min)** : tranche le point ouvert du PRD Annexe A — un vrai compte élève permet les policies RLS par utilisateur, le plus-addressing évite de créer une boîte mail, le code court garde la connexion faisable par un ado. Le wizard `/config` crée les deux comptes d'un coup et laisse l'appareil connecté en élève.
- **2026-07-04 — Sync write-through, localStorage d'abord** : chaque écriture va au localStorage (l'app marche hors ligne) puis est poussée vers Supabase ; les push ratés vont dans une file (`klaar.pushqueue.v1`) rejouée au démarrage. Au login, migration unique des données locales pré-connexion, puis le serveur fait référence (pull remplace le cache). Si un autre utilisateur s'était déjà synchronisé sur l'appareil, pas de migration (ses données ne sont pas celles du nouveau connecté).
- **2026-07-04 — Confirmation d'email à désactiver sur le projet Supabase hébergé** : le wizard crée deux comptes à la suite et a besoin des sessions immédiatement ; s'il reçoit `session: null`, il affiche la marche à suivre. En local c'est déjà désactivé par défaut.
- **2026-07-04 — Stats de session non persistées en cours de route** : si l'élève quitte ou recharge en pleine session, la file est recomposée au retour et les compteurs de la session repartent de zéro. L'état SRS par carte, lui, est sauvegardé après chaque réponse — aucune progression d'apprentissage n'est perdue.
- **2026-07-05 — Drills de grammaire : QCM à choix unique, correct = « Réussi » (qualité 5), faux = « À revoir » (qualité 2, carte re-mise en fin de file)** : la réponse d'un drill est objective, pas d'auto-évaluation à 3 boutons comme en vocabulaire ; le bouton « Difficile » n'a pas d'équivalent naturel en QCM et est omis. Même moteur SM-2, mêmes états SRS (`srs_state` commun), sessions séparées par modalité.
- **2026-07-05 — Options des drills dans `content_items.choices` (jsonb, bonne réponse incluse)** : colonne nullable ajoutée par migration plutôt qu'un encodage dans `back` — reste requêtable, extensible aux futurs QCM d'écoute (M3). L'UI mélange l'ordre à l'affichage.
- **2026-07-05 — `sessions.module` (vocab | grammar, null = vocab d'avant M2)** : permet le taux de réussite par compétence du dashboard parent (PRD §9) sans table `attempts` pour l'instant.
- **2026-07-05 — 41 drills de grammaire A2 générés (présent, hebben/zijn, négation, inversion, perfectum, modaux, pronoms, pluriel, interrogatifs)** : même statut que le vocabulaire de départ — à faire valider par un néerlandophone (PRD §13).
- **2026-07-05 — `supabase/generate-seed.mjs` régénère `seed.sql` depuis `src/data/*.json`** : une seule source de vérité pour le contenu de départ, upsert `on conflict (id)` pour pouvoir re-seeder l'hébergé sans doublons.
- **2026-07-05 — Audio de la compréhension orale : synthèse vocale du navigateur (Web Speech API, voix `nl`, débit 0,9)** : gratuite, sans clé API ni coût récurrent, disponible sur mobile ; qualité suffisante pour de l'A2. `content_items.audio_url` reste dans le schéma pour brancher de vrais enregistrements (Karrewiet, etc.) plus tard. Si aucune voix TTS n'est disponible sur l'appareil, l'app affiche le transcript à la place (l'item reste jouable).
- **2026-07-05 — Format des items d'écoute : `front` = transcript NL (lu en TTS, caché), `question` = question de compréhension en FR (nouvelle colonne), `choices`/`back` = QCM en FR** : réutilise la mécanique de drill de M2 ; le transcript est révélé après la réponse (pédagogie : relier son et écrit). Question en FR pour tester la compréhension, pas la lecture.
- **2026-07-05 — 24 items d'écoute A2 générés** : même statut que le reste du contenu de départ — à faire valider par un néerlandophone (PRD §13).
- **2026-07-05 — Badges calculés côté client, à la fin de chaque session, par des règles pures (`src/lib/badges.ts`)** : pas de trigger serveur — les données nécessaires (sessions, XP, streak) sont déjà locales, et la table `badges` (append-only, unique par code, upsert `ignoreDuplicates` pour le multi-appareils) empêche les doublons. 9 badges de lancement : premier pas, sans faute (≥ 5 cartes), menu complet (3 modalités le même jour), 25 sessions, streak 3/7/14, XP 500/1000.
- **2026-07-05 — Célébration des nouveaux badges via l'état de navigation du router** : perdu au rechargement du bilan, mais le badge est déjà persisté — seul l'effet de fête est manqué, pas le badge.
- **2026-07-09 — Type de contenu `writing` au lieu du `writing_prompt` du PRD §7** : garde l'invariant type = modalité utilisé partout (itemsForModule, `sessions.module`) ; `speaking` normalisé de la même façon dans la contrainte en vue de M5.
- **2026-07-09 — Format des items de rédaction : `front` = consigne FR avec points à couvrir (puces, rendu pre-line), `checklist` = points d'auto-évaluation FR (nouvelle colonne jsonb), `back` = texte modèle NL ~60 mots** : pas de juge automatique (PRD §13) — après rédaction, l'élève coche la checklist, voit une note suggérée non contraignante (tout coché = réussi, ≥ moitié = difficile, sinon à revoir) et s'auto-évalue avec les 3 boutons SM-2 habituels ; l'exemple de réponse est repliable pour inciter à s'évaluer avant de comparer.
- **2026-07-09 — Session rédaction = 2 textes maximum (format examen PRD §16 : deux rédactions ~60 mots), bouton « J'ai terminé » actif à partir de 40 mots** : compteur de mots visible avec l'objectif 60 ; en dessous de 40 le texte est trop court pour être évalué.
- **2026-07-09 — XP production : 25 × difficulté (vs 10 en reconnaissance)** : première application de la pondération par modalité annoncée au M1 (PRD §8 : la production rapporte plus) ; une session rédaction complète ≈ 70 XP.
- **2026-07-09 — Badge « Première rédaction » (✍️) ; description du « Menu complet » généralisée à « trois modalités différentes le même jour »** : la règle (≥ 3 modalités) reste inchangée maintenant qu'il en existe 4.
- **2026-07-09 — Streak : seuil validé par Pierre à 60 min/jour (`STREAK_MIN_MINUTES_PER_DAY`)** : conforme au PRD §8 (≥ 1 h/jour), atteignable avec 4 modalités. Pas d'effet rétroactif : aucun streak en cours sur l'hébergé (dernière activité réelle le 5 juillet). Le tooltip de l'accueil affiche les minutes restantes du jour.
- **2026-07-09 — 12 consignes de rédaction A2- générées (présentation, loisirs, vacances, météo, santé, nourriture, achats ×2, itinéraire, sortie, école, week-end au perfectum)** : même statut que le reste du contenu de départ — à faire valider par un néerlandophone (PRD §13).
- **2026-07-09 — Oral (M5) : MediaRecorder + auto-évaluation par checklist (partagée avec la rédaction), prononciation de référence = TTS de l'exemple NL** : pas de reconnaissance vocale juge (PRD §13) — l'élève écoute un exemple, s'enregistre (min 20 s, max 3 min, refaire possible), se réécoute et coche la checklist. Sans micro ou permission refusée : entraînement à voix haute, item jouable quand même (même esprit que le fallback TTS de l'écoute).
- **2026-07-09 — Enregistrements dans Supabase Storage (décision Pierre) : bucket privé `recordings`, chemin `{user_id}/{horodatage}-{item}.webm`, policies RLS par dossier racine, 10 Mo max et MIME audio uniquement** : l'élève écrit/lit/supprime son dossier, le parent du foyer écoute (créneau hebdo PRD §13). Upload best-effort à la notation, sans file de retry — les blobs feraient déborder le quota localStorage ; une prise perdue ne bloque jamais la session.
- **2026-07-09 — Rétention des enregistrements : 14 jours (décision Pierre : limiter le quota), nettoyée côté client au démarrage** : l'élève supprime ses propres fichiers expirés à l'ouverture de l'app (la policy RLS ne l'autorise que sur son dossier) — pas de tâche serveur à maintenir ; ~2 prises/jour ≈ 45 Mo par fenêtre de 14 jours, très loin des quotas.
- **2026-07-09 — Session oral = 2 sujets maximum (comme les deux parties de l'examen), plafond factorisé avec la rédaction (`PRODUCTION_CAP`)** : XP production identique (25 × difficulté), badge « Première prise de parole » 🎤.
- **2026-07-09 — 12 sujets d'oral A2- générés (présentation, famille, loisirs, école, journée, météo, nourriture, achats, docteur en jeu de rôle, itinéraire, vacances, week-end au perfectum)** : à faire valider par un néerlandophone (PRD §13).
- **2026-07-09 — Examens blancs (M6) calibrés sur les épreuves officielles 2022-2026 mais avec un contenu original** : les PDF/audio téléchargés par Pierre (`docs/CE1D/`, ignoré par git — volumineux, non redistribuable sur un repo public) ont fourni la structure et le barème (docs/CE1D-FORMAT.md : CA /30, CL /20, EE /20, EO /30, seuil 50 %) ; les épreuves de l'app sont générées au même format. Les PDF officiels servent aux répétitions papier en conditions réelles.
- **2026-07-09 — Définitions d'épreuves dans `src/data/exams.json` (pas dans `content_items`)** : un examen est une structure multi-documents (sections, tâches, corrigés, grilles pondérées) qui ne rentre pas dans le modèle carte ; seuls les RÉSULTATS sont en base (`mock_exams`, append-only, lecture parent). L'import d'épreuves du foyer viendra si le besoin apparaît.
- **2026-07-09 — Correction des blancs par auto-correction (PRD §13, pas de juge automatique)** : CA/CL = corrigé à cocher (« j'ai noté cette info »), prorata arrondi au demi-point comme la conversion officielle ; EE/EO = grille officielle simplifiée en checklist pondérée /10 ou /15, plafond 40 % si texte < 55 mots (grille n°2 officielle). Le chrono ne court que pendant la phase de réponse (CA 25 / CL 40 / EE 50 min — EE est la durée officielle, CA/CL des approximations, le guide de passation n'étant pas dans les téléchargements).
- **2026-07-09 — Boss battle (PRD §8) : +100 XP de complétion, +100 de bonus à partir de 50 %** : hors plafond de session (un examen n'est pas une session d'entraînement). L'examen enregistre aussi une session `module='exam'` à 0 carte : les minutes comptent pour le streak sans fausser les taux de réussite.
- **2026-07-09 — Rapport hebdo parent (PRD §9) généré côté client (`src/lib/report.ts`)** : 7 derniers jours vs 7 précédents (jours travaillés/validés, minutes, taux, point faible ≥ 5 cartes), pas de tâche serveur ni d'email — le parent le consulte dans le dashboard.
- **2026-07-09 — Invitation co-parent par lien (demande Pierre, PRD §3/§9)** : `/config?invite=<id du foyer>` — l'id du foyer (uuid non devinable) fait office de code d'invitation, conformément au design M1 (« rejoindre un foyer = connaître son uuid »). Aucune migration : `profiles_insert_self` autorise déjà l'insertion de son propre profil dans un foyer dont on connaît l'id, et toutes les policies de lecture parent passent par `is_parent_of` (n'importe quel parent du foyer). Le parent connecté voit le lien dans /config (copie presse-papiers) ; le co-parent crée son compte (prénom/email/mot de passe) et obtient le rôle parent. Limites assumées : le lien ne périme pas (le révoquer = pas prévu à l'échelle d'une famille) et quiconque connaît l'uuid — y compris l'élève — pourrait créer un compte « parent » ; échelle familiale, risque accepté et documenté.
- **2026-07-09 — Charte « Oranje Trainer » implémentée (piste A validée par Pierre, docs/IDENTITE.md)** : tokens Tailwind v4 sémantiques (`action` oranje = seule couleur d'action, `reward` geel = récompense uniquement, `ink` = neutres marine — renommage global teal/amber/slate) ; typo Bricolage Grotesque (display) + Instrument Sans + Atkinson Hyperlegible auto-hébergées (Fontsource) ; navigation par onglets bas sur mobile / rail latéral sur lg (masquée en session et examen : le focus est une feature) avec pages dédiées Examens et Progrès ; anneau d'objectif quotidien à l'accueil ; icônes Lucide (fin des emojis de navigation) ; largeurs par route (session étroite partout, dashboard parent en grille 2 col jusqu'à 4xl) ; motion limitée aux 3 moments de la charte sous `motion-safe` ; accessibilité M7 (taille de texte 3 crans + police Hyperlegible, préférences locales). NB : les scripts Playwright qui lisent l'XP au bilan doivent attendre ~1,3 s (compteur animé).
- **2026-07-09 — Retouches charte (feedback Pierre)** : logo bicolore (« Klaar » encre, « ! » oranje) décliné au favicon et au theme-color ; la pastille d'état déménage de l'accueil vers une chip compacte du header (prénom / Démo / Local, à côté de la bascule de thème) ; le parent connecté est redirigé de l'accueil vers /parent et ses onglets deviennent Suivi / Importer / Profil ; en mode démo/local, « Recommencer à zéro » dans le profil efface les données d'apprentissage de l'appareil (conserve thème et préférences d'affichage) et ramène à l'écran visiteur.
- **2026-07-09 — Renommage du parent (demande Pierre) : grant UPDATE limité à la colonne `display_name` + policy « sa propre ligne, rôle parent »** : deux couches — le grant par colonne empêche tout changement de `role` ou de `household_id` (même sur sa propre ligne), la policy empêche de renommer quelqu'un d'autre et exclut l'élève (son prénom est fixé par le parent à la création). UI : crayon ✏️ à côté du nom dans /config, la pastille de l'accueil se met à jour via le cache profil local.
- **2026-07-09 — État de connexion visible à l'accueil + mode démo (feedback Pierre, PRD §17)** : pastille permanente en haut de l'accueil (👤 prénom · rôle si connecté, 🧪 « Mode démo » sinon, 📱 « Mode local » si le déploiement n'a pas de Supabase), cliquable vers /config. Visiteur sans compte ni choix préalable → écran de bienvenue « Se connecter / Essayer en mode démo » (flag `klaar.demo.v1`). Des données locales préexistantes valent choix implicite du mode démo (pas de régression pour les appareils déjà utilisés) ; à la première connexion, la migration locale existante reprend les données démo dans le compte.
